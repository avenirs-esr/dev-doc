## Introduction

La Definition of Ready (DoR) définit les conditions minimales qu’une User Story (US) doit remplir pour être prise en charge en développement.

Une User Story **Ready** est :
- comprise par l’équipe
- suffisamment précise pour être développée
- testable
- démarrable sans blocage majeur

---

## Critères de Ready

Une User Story est considérée comme **Ready** si les conditions suivantes sont remplies :

### 1. Refinement réalisé

- La User Story a été présentée et discutée lors d’une séance de refinement
- Les questions identifiées ont reçu une réponse ou un plan d’action est défini

---

### 2. Valeur métier explicitée

- La valeur métier ou utilisateur est clairement décrite
- L’équipe comprend le besoin auquel répond la User Story

---

### 3. Critères d’acceptation définis

- Des critères d’acceptation sont présents
- Ils sont compréhensibles, précis et testables

---

### 4. Éléments de conception disponibles

- Les maquettes, wireframes ou éléments Figma nécessaires sont disponibles lorsque requis

---

### 5. Compréhension commune

- L’équipe de développement estime disposer des informations nécessaires pour démarrer les travaux
- Les éventuels risques ou points techniques majeurs ont été identifiés

---

## Non objectifs de la DoR

La DoR ne couvre pas :
- les exigences de qualité (Definition of Done)
- les validations CI / sécurité / performance
- le pilotage projet (priorité, sizing, dépendances)

Ces éléments sont définis dans :
- la Definition of Done (DoD)
- le GitHub Project

---

## Principe

Une User Story n’a pas besoin d’être parfaite pour être considérée comme Ready.

Elle doit être suffisamment claire pour permettre à l’équipe de commencer son développement sans blocage majeur et sans ambiguïté fonctionnelle.
